# Routing Waterway System (RWS)

## Overview
RWS — система построения маршрутов по внутренним водным путям между двумя географическими точками.

Пользователь задаёт:
- координату A
- координату B
- время отправления
- скорость движения

Система возвращает:
- общее расстояние маршрута
- расчётное время в пути
- расчётное время прибытия (ETA)
- список шлюзов на маршруте
- геометрию маршрута для отображения на карте

Маршрут может строиться:
- между произвольными координатами
- между портами
- между любыми точками, принадлежащими водной сети

## Architecture
Система построена на микросервисной архитектуре.

### Сервисы:
1) rws-api
- Внешний REST API (HTTP).
- Выполняет роль API Gateway.
- Принимает пользовательские запросы и агрегирует ответы.
2) route-api
- Сервис построения маршрута.
- Выполняет алгоритм поиска пути
- Вычисляет расстояние, ETA
- Определяет шлюзы на маршруте
- Взаимодействует с refdata-api по gRPC для получения портов и шлюзов
3) refdata-api
* Предоставляет справочную информацию: порты, шлюзы, дополнительные атрибуты объектов
4) geography-importer
- Импортирует географические данные (PBF) в базу geography.
5) archvie-api
* Работа с архивными перевозками
* Дает информацию (статистику) по уже проделанным маршрутам
* Взаимодействует с route-api для воспроизведения маршрута

## Service Communication
- Внешний доступ — REST (через rws-api)
- Межсервисное взаимодействие — gRPC
- Сервисы логически изолированы

Инварианты архитектуры:
- route-api не зависит от доступности refdata-api для построения маршрута
- refdata-api полностью независим от route-api
- отказ одного сервиса не приводит к полной деградации системы

## Data Storage
Используется PostgreSQL 16.

### Databases
1) geography: ноды, рёбра, геометрия, граф навигационной сети
2) refdata: порты, шлюзы, справочные сущности

Используемые расширения:
- PostGIS
- pgRouting

## Routing Algorithm
Маршрут строится с использованием алгоритма A*.
Алгоритм выполняется на уровне базы данных (PostgreSQL) с использованием pgRouting и PostGIS.

Принципы:
- граф хранится в geography
- вес рёбер определяется длиной и атрибутами навигации
- расчёт производится SQL-функциями
- вычисление выполняется атомарно внутри транзакции

## Caching
Используется Redis.

Назначение:
- кэширование недавно построенных маршрутов
- ускорение повторных запросов
- снижение нагрузки на базу geography

## Observability
Используются:
- Prometheus — метрики
- Loki — логирование
- Tempo — трассировка
Поддерживается распределённая трассировка между сервисами.

## Architectural Constraints
- Алгоритм маршрутизации выполняется на стороне БД, а не в Java
- Сервисы не разделяют общую БД
- Взаимодействие строго через gRPC
- Redis не является источником истины

# Project status
Система находится в стадии разработки.
Формальный SLA отсутствует.

Целевые инженерные показатели:
- расчёт типового маршрута ≤ 1 секунды
- изоляция сервисов
- отсутствие общей БД между сервисами
- возможность горизонтального масштабирования

## Constraints
На текущий момент система:
- не учитывает погодные условия
- не учитывает расписание шлюзования
- не выполняет динамическую оптимизацию скорости

## Domain Assumptions
- Граф навигации представляет только внутренние водные пути.
- Ребра представляют собой судоходные участки воды.
- Шлюзы моделируются как специальные объекты, связанные с участками маршрута.
- Расчетное время прибытия (ETA) производится на основе расстояния и постоянной скорости судна.
- Динамические изменения трафика, погоды и расписания шлюзов не учитываются.
